**Структура Backend**

Backend побудований на стандартній структурі Laravel із дотриманням принципів розділення відповідальності та чистої архітектури.

Збережена стандартна структура Laravel, використовуються invokable контролери, service файли та Eloquent ORM для виконання запитів до бази даних.

Для усунення хардкоду використовується:
* файли перекладів
* файли з enum-константами

Для збереження цілісності даних застосовується DTO (Data Transfer Objects)

**Підхід**
* Контролери максимально “тонкі” і відповідають лише за виклик сервісів
* Вся бізнес-логіка винесена в service layer
* Робота з БД реалізована через Eloquent ORM
* Валідація та форматування відповіді винесені в окремі класи (Request та Resources)
* Для уніфікації та обмеження допустимих значень використовуються enum-класи
* Для передачі та захисту структури даних між шарами використовується DTO (Data Transfer Objects)

**Структура Frontend**

Frontend побудований на Nuxt 4 і розділений на логічні шари для забезпечення масштабованості, повторного використання коду та чистої архітектури.

Pages (сторінки)

Відповідають за маршрутизацію та відображення основних екранів застосунку:
* список інвойсів
* перегляд інвойсу
* редагування інвойсу

**Компроміси**

* Backend реалізований без тестового покриття.
* Відсутнє наповнення бази даних тестовими даними автоматичне початкове наповнення (database seeding)
* Не використовується L5 Repository
* У Frontend не використовується винесення коду в окремі компоненти, оскільки сторінки не містять великого обсягу логіки або розмітки.
* Не використовується store (хранилище стану), оскільки стан обмежений межами сторінок і не потребує глобального управління.
* Backend та Frontend знаходяться у одному проекті

**Покращення**

* Для production-версії виправив би наявні компроміси
* Додав би API-документацію
* Додав би на фронтенді сторінку створеня інвойса

**UX edge cases***

У процесі розробки були враховані типові UX edge cases для забезпечення стабільної та передбачуваної поведінки інтерфейсу:

* Обробка станів loading / error / empty state під час запитів до API
* Блокування форми редагування, якщо статус інвойсу не дозволяє зміни (status !== pending)
* Захист від некоректного вводу через валідацію (Zod)
* Автоматичний перерахунок gross_amount без ручного введення користувачем
* Відображення fallback-значень (—) для відсутніх полів
* Перенаправлення після успішного оновлення даних із затримкою для UX-зворотного зв’язку
* Обробка помилок запиту з інформуванням користувача про невдалу операцію
